## Implementation Steps

### 1. Set up the test environment

Open Chrome DevTools with 4x CPU throttle enabled (Performance tab -> CPU -> 4x slowdown). This simulates approximate mobile device performance. Disable cache to measure real load behavior.

### 2. Profile the full user flow

Walk through the complete flow: landing page -> enter editor -> select assets for all slots -> global adjust -> scene decorate -> music selection -> render -> share. At each stage, record:

- Peak JS heap memory (Performance tab -> Memory)
- FPS during 3D interaction (Performance tab -> Frames)
- Time to complete each transition

### 3. Measure render pipeline performance

Start a 10-second video render and measure:

- Total render time (wall-clock seconds)
- Peak memory during render
- Whether the browser tab remains responsive during render

### 4. Check for memory leaks

Perform the following repeatedly (3+ cycles): load an asset, unload it, load a different asset. Take heap snapshots between cycles and compare. If memory grows monotonically, identify the leaking objects using the "Comparison" view in the Memory tab.

### 5. Evaluate against targets

Compare measurements to the following targets:

- Peak memory < 300 MB during editing
- 30+ fps during 3D interaction in the editor
- Render time < 60 seconds for a 10-second video (under 4x CPU throttle)
- No memory increase on repeated asset load/unload cycles

### 6. Identify and apply optimizations

If targets are not met, use the Chrome Performance profiler to identify bottlenecks. Common fixes include: reducing texture resolution, adding LODs for complex models, limiting the number of concurrently loaded assets, simplifying shaders, and disposing Three.js resources explicitly.

### 7. Document findings

Create a performance report with actual measurements, any optimizations applied, and remaining known issues. Include device/browser details for reproducibility.
