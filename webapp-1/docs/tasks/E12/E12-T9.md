## Implementation Steps

### 1. Detect return from Stripe Checkout

When the user returns from Stripe Checkout, the success URL includes `?success=1`. Detect this query parameter on page load or route change.

### 2. Verify entitlement with retry

Call `GET /api/v1/entitlements/{sessionId}` to confirm the entitlement was created. If the webhook has a slight delay:

1. Retry with exponential backoff: 3 attempts at 1 s, 2 s, then 4 s intervals.
2. If entitled after any retry: update Zustand `entitlementSlice`, dismiss the paywall, and enable share/save.
3. If still not entitled after all retries: show a message "Payment processing, please wait..." with a manual retry button.

### 3. Handle cancelled payment

If the user returns with `?canceled=1`, show the paywall again or return to the video preview with export still blocked.

### 4. Clean up URL parameters

After handling `success` or `canceled` params, remove them from the URL using `history.replaceState` so that refreshing the page does not re-trigger the logic.

### 5. Handle page refresh on success

If the user refreshes the success page, re-run the entitlement check (the URL params will already be cleaned, but the session ID in storage still allows verification).

## Completion Criteria

- [ ] After Stripe payment, returning to the app successfully verifies the entitlement
- [ ] Export buttons become active after verification
- [ ] Slight webhook delay handled gracefully (retry logic works)
- [ ] Cancelled payment returns the user to video preview with paywall available
- [ ] URL is cleaned up after handling (no lingering `?success=1`)
- [ ] Works even if the user refreshes the success page
