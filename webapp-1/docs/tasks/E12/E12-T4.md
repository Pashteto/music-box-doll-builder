## Implementation Steps

### 1. Create the endpoint

`POST /api/v1/entitlements/restore`

Request body: `{ email: string }`

### 2. Look up entitlements by email

Query PostgreSQL for entitlements where `email` matches (case-insensitive) and `status = 'active'`.

Email comparison must be case-insensitive — use `LOWER(email)` or a case-insensitive collation.

### 3. Return the result

If found: `{ entitled: true, sessionId: originalSessionId }` — the frontend can then re-link the entitlement to the current session.

If not found: `{ entitled: false, message: "No purchase found for this email" }`

### 4. Apply aggressive rate limiting

Rate-limit this endpoint more aggressively than others to prevent email enumeration attacks.

### 5. Prevent information leakage

Do not reveal whether a specific email exists in the system through error responses or timing differences. Return the same generic response for any "not found" case.

**Note:** The `email` field is only populated when the user provides their email during Stripe checkout or during restore. Phase 1 purchase is anonymous, so restore only works if the user provided their email during the Stripe Checkout flow.

## Completion Criteria

- [ ] Known email with active entitlement returns `{ entitled: true }`
- [ ] Unknown email returns `{ entitled: false }` with a message
- [ ] Email lookup is case-insensitive
- [ ] Rate limiting prevents rapid-fire requests
- [ ] No email addresses leaked in error responses
