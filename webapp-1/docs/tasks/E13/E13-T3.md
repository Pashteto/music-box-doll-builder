## Implementation Steps

### 1. Create the analytics handler

Create `internal/analytics/handler.go`. Register the route `POST /api/v1/analytics/events` in the main router.

### 2. Define the request schema

Accept a JSON body with the following structure:

```go
type EventBatch struct {
    Events []Event `json:"events"`
}
```

Each `Event` contains: `Event` (string, required), `Properties` (map, optional), `Timestamp` (ISO-8601 string, required).

### 3. Validate incoming batches

- Reject batches with more than 50 events (return 400)
- Reject events missing required fields (`event`, `timestamp`) (return 400)
- Validate that `timestamp` is a valid ISO-8601 datetime

### 4. Forward events to PostHog

For Phase 1, forward the batch to PostHog's server-side `/capture` endpoint using an HTTP POST. Include the PostHog project API key from the environment variable `POSTHOG_API_KEY`. This server-side forwarding bypasses client-side ad blockers.

### 5. Implement rate limiting

Rate-limit requests per `sessionId` (extracted from the session cookie or from a header). Use an in-memory rate limiter (e.g., `golang.org/x/time/rate`) or Redis-backed limiter. A reasonable limit is 10 requests per minute per session.

### 6. Handle PostHog unavailability

If the PostHog API is unreachable or returns an error, log the failure and return 202 (accepted) to the client. Do not block the client on PostHog availability. Optionally queue failed batches for retry (out of scope for Phase 1 — logging is sufficient).

## Completion Criteria

- [ ] `POST /api/v1/analytics/events` with a valid batch returns 200
- [ ] Events forwarded to PostHog appear in the PostHog dashboard
- [ ] Batch exceeding 50 events returns 400 with an error message
- [ ] Events missing required fields return 400
- [ ] Rate limiting works — exceeding 10 req/min per session returns 429
- [ ] If PostHog is unavailable, the endpoint returns 202 and logs the error (does not crash or hang)
- [ ] Endpoint requires no authentication (anonymous analytics)
- [ ] `POSTHOG_API_KEY` is documented in `backend/.env.example`
