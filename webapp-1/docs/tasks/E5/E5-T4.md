## Implementation Steps

### 1. Implement proper disposal pattern

Create a `useDispose` hook or utility function at `src/modules/scene/useDispose.ts`. When an asset is removed or replaced, traverse the GLTF scene graph and call `dispose()` on every geometry, material, and texture. This prevents GPU memory leaks.

The utility should recursively walk `object.traverse()` and handle:

- `mesh.geometry.dispose()`
- `material.dispose()` (including array materials)
- `material.map?.dispose()`, `material.normalMap?.dispose()`, etc.

### 2. Integrate disposal into the component lifecycle

When a `SlotMesh` unmounts (asset replaced or slot cleared), call the dispose utility on the old GLTF scene in a `useEffect` cleanup function. Verify that React's unmount triggers disposal correctly.

### 3. Cap renderer pixel ratio

In the `DollScene` Canvas config (E5-T1), ensure the pixel ratio is set to `Math.min(window.devicePixelRatio, 2)`. This is critical for 3x retina displays (iPhone Pro) where rendering at native resolution would triple GPU workload with minimal perceptible quality gain.

### 4. Verify frustum culling

Three.js enables frustum culling by default. Verify this by confirming that `mesh.frustumCulled === true` on loaded assets. If any asset has it disabled, re-enable it. Objects outside the camera view should not be rendered.

### 5. Monitor renderer.info during development

Access `renderer.info` (via `useThree`) and periodically log counts of geometries, textures, and shader programs. Create a simple dev-only overlay or console logger (gated behind `process.env.NODE_ENV === 'development'`) that shows these numbers. This helps catch leaks early.

### 6. Add FPS counter for development

Add Drei's `<Stats>` component (or a custom implementation) inside the Canvas, visible only in development mode. This provides real-time FPS, MS per frame, and memory stats during testing.

### 7. Profile on simulated mobile

Test the scene with 5 loaded GLB placeholder assets simultaneously. In Chrome DevTools, enable 4x CPU throttling to simulate mobile performance. The scene should maintain 30+ fps while orbiting. Document the profiling results and any optimizations applied.

## Completion Criteria

- [ ] Loading then removing an asset does not leak geometries or textures (verify via `renderer.info` counts returning to baseline)
- [ ] 5 placeholder assets loaded simultaneously maintain 30+ fps on Chrome with 4x CPU throttle
- [ ] Memory usage with 5 assets loaded stays well under 300 MB (check DevTools Memory tab)
- [ ] No "WebGL context lost" errors during a full editing session
- [ ] FPS counter (dev mode only) shows a stable framerate during orbit interaction
- [ ] `useDispose` correctly cleans up all geometry, material, and texture resources
- [ ] Pixel ratio is confirmed clamped to max 2 on high-DPI devices
- [ ] Frustum culling is active on all loaded meshes
