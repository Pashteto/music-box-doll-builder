## Implementation Steps

### 1. Create the session middleware

Create `internal/auth/session.go`. Implement an HTTP middleware function that runs on every request (except excluded paths).

### 2. Implement session cookie logic

The middleware should:

- **Cookie present:** Extract the session UUID from the `doll_session_id` cookie, verify it exists in the PostgreSQL sessions table, and attach it to the request context
- **Cookie absent:** Generate a new UUID (v4), insert a row into the sessions table, and set the `doll_session_id` cookie on the response

### 3. Configure cookie attributes

Set the cookie with the following attributes:

- `HttpOnly: true` (not accessible to JavaScript)
- `Secure: true` (only sent over HTTPS)
- `SameSite: Lax` (protects against CSRF while allowing top-level navigation)
- `Max-Age: 31536000` (1 year)
- `Path: /`

### 4. Exclude health and webhook endpoints

Do not require a session cookie for:

- `GET /api/v1/health`
- `POST /api/v1/stripe/webhook`

These endpoints should bypass the session middleware entirely.

### 5. Configure CORS for frontend origin

Set up CORS middleware to:

- Allow the frontend origin (from environment variable `FRONTEND_URL`)
- Allow credentials (`Access-Control-Allow-Credentials: true`) so cookies are sent cross-origin
- Allow required headers (`Content-Type`, etc.)
- Allow required methods (`GET`, `POST`, `OPTIONS`)

### 6. Document frontend-side backup

The frontend should also store the session UUID in `localStorage` as a backup (cookies can be cleared by users or browsers). This is documented here for coordination but implemented on the frontend side (E12-T7's sessionId module).

## Completion Criteria

- [ ] First request to any API endpoint (except excluded ones) sets a `doll_session_id` cookie
- [ ] Subsequent requests include the cookie, and the session is found in PostgreSQL
- [ ] Session UUID is a valid UUID v4 format
- [ ] Cookie has `HttpOnly`, `Secure`, and `SameSite=Lax` attributes
- [ ] Health endpoint and Stripe webhook endpoint work without a session cookie
- [ ] CORS headers allow the frontend origin with credentials
- [ ] Same session persists across multiple API calls (cookie is not regenerated on each request)
- [ ] Invalid session UUID in cookie triggers creation of a new session (not a 500 error)
- [ ] `FRONTEND_URL` is documented in `backend/.env.example`
