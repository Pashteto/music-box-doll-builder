## Implementation Steps

### 1. Create the catalog handler

Create `internal/catalog/handler.go`. Register the route `GET /api/v1/catalog/manifest` in the main router.

### 2. Serve the manifest JSON

For Phase 1, serve the catalog manifest JSON directly from the Go server. The manifest can be:

- Embedded in the binary using Go's `embed` package (recommended for simplicity)
- Read from the filesystem at a configurable path

Load the manifest once at startup and serve it from memory on each request.

### 3. Set cache headers

Set the `Cache-Control` header to `public, max-age=3600` (1 hour). This allows browsers and CDNs to cache the manifest and reduce backend load. Adjust the duration based on how frequently the catalog changes.

### 4. Implement conditional request support

Generate an `ETag` header from the manifest content hash (e.g., SHA-256 of the JSON bytes). On incoming requests, check the `If-None-Match` header. If it matches the current ETag, return `304 Not Modified` with no body.

### 5. Ensure GZIP compression

GZIP compression should be handled by the HTTP middleware layer (e.g., chi middleware or a dedicated compression middleware). Verify that `Content-Encoding: gzip` is present in responses to clients that send `Accept-Encoding: gzip`.

### 6. Plan for Phase 2 evolution

Add a comment in the handler noting that in Phase 2, this endpoint will become dynamic and serve personalized catalogs based on user entitlements. For now, all users receive the same manifest.

## Completion Criteria

- [ ] `GET /api/v1/catalog/manifest` returns valid JSON matching the CatalogManifest type
- [ ] Cache headers are set correctly (`Cache-Control: public, max-age=3600`)
- [ ] Response is GZIP compressed (check `Content-Encoding` header)
- [ ] `ETag` header is present in the response
- [ ] Conditional request with `If-None-Match` returns 304 when the manifest is unchanged
- [ ] Response time < 50 ms (serving static JSON from memory)
- [ ] Endpoint works without authentication (public catalog)
