## Implementation Steps

### 1. Create workflow file

Create `.github/workflows/ci.yml` at the repository root.

### 2. Configure triggers

Set the workflow to trigger on:

- `push` to `main` branch
- `pull_request` targeting `main` branch

### 3. Define the frontend job

Job name: `frontend`

Steps:
1. **Checkout** — `actions/checkout@v4`
2. **Setup Node** — `actions/setup-node@v4` with `node-version: 20`
3. **Cache node_modules** — Use `actions/cache@v4` with key based on `package-lock.json` hash, path `~/.npm` (or `node_modules/`)
4. **Install** — `npm ci`
5. **Lint** — `npm run lint`
6. **Build** — `npm run build`

### 4. Define the backend job

Job name: `backend`

Set `defaults.run.working-directory` to `backend/`.

Steps:
1. **Checkout** — `actions/checkout@v4`
2. **Setup Go** — `actions/setup-go@v5` with `go-version: '1.22'`
3. **Cache Go modules** — Use `actions/cache@v4` with key based on `go.sum` hash, paths `~/go/pkg/mod` and `~/.cache/go-build`
4. **Vet** — `go vet ./...`
5. **Test** — `go test ./...`
6. **Build** — `go build ./cmd/api`

### 5. Add placeholder deploy steps

Add commented-out deploy steps at the end of each job:

- **Frontend:** Placeholder for Vercel deployment (e.g., `vercel deploy --prod`)
- **Backend:** Placeholder for container registry push and deploy (e.g., `docker build`, `docker push`, `flyctl deploy`)

Mark these with `# TODO: Enable when deployment targets are configured`.

### 6. Validate the workflow

Before pushing, validate the YAML syntax. Options:

- Use `actionlint` locally if installed (`brew install actionlint && actionlint .github/workflows/ci.yml`)
- Push to a feature branch and verify the workflow appears in the GitHub Actions tab
- At minimum, ensure the YAML parses correctly with a YAML validator

## Completion Criteria

- [ ] `.github/workflows/ci.yml` exists and is valid YAML
- [ ] Workflow triggers on push to `main` and pull requests to `main`
- [ ] `frontend` job runs: checkout, Node 20 setup, npm ci, lint, build
- [ ] `backend` job runs: checkout, Go setup, go vet, go test, go build
- [ ] Both jobs use caching (node_modules / npm cache for frontend, Go modules for backend)
- [ ] Commented-out deploy placeholders are present for both frontend and backend
- [ ] Both jobs would pass given the current clean state of both projects (no lint errors, no build failures)
