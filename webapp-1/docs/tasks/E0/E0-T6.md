## Implementation Steps

### 1. Create docker-compose.yml

Create `docker-compose.yml` at the project root with three services:

**postgres:**
- Image: `postgres:16-alpine`
- Exposed port: `5432`
- Environment: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` (read from `.env`)
- Volume: named volume `pgdata` mounted to `/var/lib/postgresql/data`
- Init script: mount `scripts/init-db.sql` to `/docker-entrypoint-initdb.d/`
- Healthcheck: `pg_isready -U $POSTGRES_USER`

**redis:**
- Image: `redis:7-alpine`
- Exposed port: `6379`
- Healthcheck: `redis-cli ping`

**api:**
- Build context: `./backend`
- Depends on: `postgres` and `redis` (with `condition: service_healthy`)
- Ports: map host `8080` to container `8080`
- Environment: read from `.env` file
- The `DATABASE_URL` and `REDIS_URL` should reference the Docker service hostnames (`postgres`, `redis`)

### 2. Create init-db.sql

Create `scripts/init-db.sql` with a placeholder schema:

```sql
-- Placeholder: schema migrations will be added later
SELECT 'Database initialized' AS status;
```

This file is mounted into the PostgreSQL container and runs on first start.

### 3. Create project-root .env.example

Create `.env.example` at the project root (separate from `backend/.env.example`) with all variables needed by docker-compose:

```
POSTGRES_USER=dollapp
POSTGRES_PASSWORD=dollapp_dev
POSTGRES_DB=dollapp
```

Include backend variables as well: `PORT`, `DATABASE_URL` (pointing to `postgres` hostname), `REDIS_URL` (pointing to `redis` hostname), `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`.

### 4. Document usage in README.md

Add a "Local Development" section to `README.md` (create the file if it does not exist) with:

- Copy `.env.example` to `.env`
- Run `docker compose up -d` to start all services
- Run `docker compose ps` to verify
- Run `docker compose down` to stop
- Note that PostgreSQL data persists in the `pgdata` volume

### 5. Verify connectivity (optional enhancement)

Consider extending the health endpoint in the Go API (from E0-T5) to check PostgreSQL and Redis connectivity. This is optional for this task but useful for verifying the full stack. At minimum, the API should start and respond on `/api/v1/health` even if DB/Redis are not yet wired in the code.
