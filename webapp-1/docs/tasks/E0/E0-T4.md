## Implementation Steps

### 1. Install Framer Motion

Install the latest stable version:

```bash
npm install framer-motion
```

Verify in `package.json` that the installed version is compatible with React 19 and Next.js 15. Check the Framer Motion changelog or GitHub releases if unsure.

### 2. Understand App Router constraints

In Next.js 15 App Router, all components are Server Components by default. Framer Motion relies on browser APIs and React hooks, so any component using `motion.*` elements **must** be a Client Component marked with `'use client'` at the top of the file.

### 3. Create a test component

Create a Client Component at `src/components/MotionTest.tsx`:

- Add `'use client'` directive at the top
- Import `motion` from `framer-motion`
- Render a `motion.div` with `initial={{ opacity: 0 }}` and `animate={{ opacity: 1 }}` and a visible background color / text so the animation is observable
- Apply a `transition` prop (e.g., `{ duration: 0.5 }`)

### 4. Mount the test component

Import and render `<MotionTest />` in `src/app/page.tsx`. Since the page itself may be a Server Component, the import of a Client Component is fine — Next.js handles the boundary automatically.

### 5. Verify no SSR hydration issues

Run `npm run dev` and open the browser console. Confirm:

- The fade-in animation plays on page load
- There are **no** hydration mismatch warnings in the console
- There are no runtime errors related to `motion` or `framer-motion`

If hydration warnings appear, ensure the `motion.div` does not depend on client-only values during initial render (e.g., `window` dimensions). The `initial` and `animate` pattern should be safe by default.

### 6. Clean up (optional)

The test component can remain as a reference for future development or be removed after verification. If kept, add a brief comment explaining it is a smoke-test for Framer Motion integration.
