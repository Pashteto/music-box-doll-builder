## Implementation Steps

### 1. Create the catalog hook file

Create `src/modules/catalog/useCatalog.ts`. This is a custom React hook that loads and caches the asset manifest.

### 2. Implement singleton caching

Use a module-level variable to store the fetched manifest:

```ts
let cachedManifest: CatalogManifest | null = null;
let fetchPromise: Promise<CatalogManifest> | null = null;
```

This ensures that no matter how many components call `useCatalog()`, only one network request is made. The cached result is reused across all consumers.

### 3. Implement the fetch logic

On first call, fetch the manifest JSON:

- In development: fetch from `/catalog/manifest.json` (served by Next.js from `public/`)
- In production: fetch from the CDN URL (configurable via environment variable, e.g., `NEXT_PUBLIC_CATALOG_URL`)

After fetching, validate the response with `isValidManifest()` from E3-T1. If validation fails, treat it as an error.

### 4. Define the hook return type

```ts
export function useCatalog(): {
  manifest: CatalogManifest | null;
  isLoading: boolean;
  error: Error | null;
}
```

Use React `useState` and `useEffect` internally. Set `isLoading` to `true` during the fetch, `false` after completion (success or failure). Set `error` on fetch failure or validation failure.

### 5. Implement helper functions

Export standalone helper functions that operate on the cached manifest (or accept the manifest as a parameter):

- `getAssetsForSlot(slotType: SlotType): AssetManifestEntry[]` -- filter the `assets` array by `slotType`
- `getAssetById(assetId: string): AssetManifestEntry | undefined` -- search all asset arrays (assets, backgrounds, foregrounds, props) for a matching ID
- `getBackgrounds(): AssetManifestEntry[]` -- return the `backgrounds` array
- `getForegrounds(): AssetManifestEntry[]` -- return the `foregrounds` array
- `getProps(): AssetManifestEntry[]` -- return the `props` array
- `getMusicTracks(): MusicTrackEntry[]` -- return the `musicTracks` array

These helpers can be pure functions that accept `CatalogManifest` as a parameter, or they can read from the module-level cache if it exists.

### 6. Handle errors and retry

If the fetch fails (network error, non-200 status, invalid JSON, validation failure):

- Set the `error` state with a descriptive message
- Allow retry by clearing the cache and re-triggering the fetch (e.g., expose a `retry()` function from the hook, or re-fetch on next mount after error)
- Never leave the hook in a permanently broken state

### 7. Write tests

Create `src/modules/catalog/__tests__/useCatalog.test.ts`:

- Mock `fetch` to return valid manifest JSON -- verify `manifest` is populated and `isLoading` transitions from true to false
- Mock `fetch` to return an error -- verify `error` is set and `manifest` is null
- Call hook twice -- verify only one fetch request is made (singleton behavior)
- Verify `getAssetsForSlot('head')` returns only head assets from the test manifest

## Completion Criteria

- [ ] `useCatalog()` returns the manifest data after loading completes
- [ ] `isLoading` is `true` during the fetch and `false` after (both success and error cases)
- [ ] `error` is set with a meaningful message on fetch failure
- [ ] `getAssetsForSlot('head')` returns only assets where `slotType === 'head'`
- [ ] `getAssetById('head-round-01')` returns the correct asset entry
- [ ] Multiple calls to `useCatalog()` across different components do not trigger multiple fetches
- [ ] Works correctly inside a React component without infinite re-renders or memory leaks
- [ ] Fetch URL is configurable between dev (local path) and prod (CDN URL) via environment variable
- [ ] Invalid manifest data from the server results in an error state, not a silent failure
