## Implementation Steps

### 1. Create the catalog types file

Create `src/lib/catalog-types.ts`. This file defines the TypeScript interfaces for the asset catalog system -- the read-only manifest data delivered via CDN.

### 2. Define the Transform type

```ts
/** Position, rotation (Euler radians), and uniform scale. */
export interface Transform {
  position: [number, number, number];
  rotation: [number, number, number];
  scale: number;
}
```

### 3. Define AssetManifestEntry

This is the core type describing a single 3D asset in the catalog. All fields match the spec's data model table exactly:

- `assetId: string` -- unique identifier
- `slotType: SlotType` -- which slot this asset belongs to (import from `types.ts`)
- `displayName: string` -- shown in the catalog UI
- `previewImage: string` -- CDN URL to a WebP thumbnail
- `glbFile: string` -- CDN URL to the Draco-compressed GLB file
- `textureFormat: 'ktx2' | 'embedded'` -- texture compression strategy
- `defaultTransform: Transform` -- initial placement when selected
- `minScale: number` -- lower bound for uniform scale
- `maxScale: number` -- upper bound for uniform scale
- `minRotation: [number, number, number]` -- lower rotation bounds per axis (radians)
- `maxRotation: [number, number, number]` -- upper rotation bounds per axis (radians)
- `anchorPoint: [number, number, number]` -- attachment point in the scene
- `excludes: string[]` -- asset IDs that are incompatible with this asset
- `dependencies: string[]` -- asset IDs that must be present alongside this asset
- `fileSizeBytes: number` -- GLB file size (for loading progress UI)
- `triangleCount: number` -- for runtime performance budgeting

Add JSDoc comments to each field explaining its purpose and expected format.

### 4. Define MusicTrackEntry

```ts
export interface MusicTrackEntry {
  trackId: string;
  displayName: string;
  audioFile: string; // CDN URL to audio file
  durationSeconds: number;
  previewImage?: string;
}
```

### 5. Define CatalogManifest

The top-level manifest object that is fetched and cached at app init:

- `version: string` -- manifest version for cache busting (e.g., "1.0.0")
- `assets: AssetManifestEntry[]` -- all slot assets (head, hair, body, etc.)
- `backgrounds: AssetManifestEntry[]` -- scene background assets
- `foregrounds: AssetManifestEntry[]` -- scene foreground assets
- `props: AssetManifestEntry[]` -- floating decorative props
- `musicTracks: MusicTrackEntry[]` -- available music tracks

### 6. Implement runtime type guard

Export a type guard function for validating manifest data fetched from the network:

```ts
export function isValidManifest(
  data: unknown
): data is CatalogManifest { ... }
```

The guard should verify:

- `data` is a non-null object
- `version` is a string
- `assets` is an array where each element has required string fields (`assetId`, `slotType`, `displayName`, `glbFile`)
- `musicTracks` is an array where each element has `trackId` and `audioFile`
- Does not need to validate every field exhaustively -- focus on structural correctness

### 7. Write tests for the type guard

Create `src/lib/__tests__/catalog-types.test.ts`:

- Valid manifest passes the guard
- Missing `version` field fails
- Missing `assets` array fails
- Asset entry missing `assetId` fails
- Empty manifest (version + empty arrays) passes
- Non-object input (string, number, null) fails
