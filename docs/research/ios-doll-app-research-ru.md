# Исследовательский бриф по iOS-приложению с куклой-конструктором

## Краткое описание проекта
Этот документ содержит исследование и рекомендации по созданию iOS-приложения, в котором пользователь сможет собирать кастомизируемую виртуальную куклу, помещать её в стилизованную сцену музыкальной шкатулки, добавлять декоративные элементы и музыку, рендерить короткое видео и делиться им вовне.

Предполагаемый пользовательский сценарий:
1. Анимированный стартовый экран
2. Пошаговый конструктор куклы
3. Режим глобальной, но ограниченной настройки
4. Декорирование и стилизация сцены
5. Выбор музыки
6. Локальный рендер / экспорт видео
7. Шеринг
8. Пейволл после первого успешного экспорта / шеринга

---

## Рекомендуемый базовый техстек

### iOS-приложение
- **Swift**
- **SwiftUI** для интерфейса и навигации
- **RealityKit** для 3D-композиции, сборки сцены и рендера
- **AVFoundation** для локальной композиции и экспорта видео
- **StoreKit 2** для покупок и подписок
- **URLSession** для API и доставки ассетов
- **BackgroundTasks** для отложенной синхронизации и фоновых загрузок
- **Нативный iOS share sheet** для экспорта и шеринга

### Бэкенд
- **Golang** для API и серверной логики
- **Redis** для кэша, очередей, rate limit, горячего состояния и краткоживущих сессий
- **PostgreSQL** для постоянных бизнес-данных
- **S3-совместимое объектное хранилище** для 3D-ассетов, превью и экспортируемых медиа
- **CDN** для быстрой доставки ассетов
- **Docker** для упаковки
- **GitHub Actions** или аналог для CI/CD
- **OpenTelemetry** для наблюдаемости

### Рекомендуемый архитектурный стиль
- Начать с **модульного монолита** на Go
- **Не** начинать с микросервисов
- Разделить внутреннюю структуру на модули:
  - auth
  - catalog
  - projects/compositions
  - entitlements
  - analytics
  - admin/content operations
  - asset manifests

## Концепция продукта
Приложение должно позволять собирать куклу из заранее определённых категорий частей, а не предоставлять пользователю полностью свободное 3D-моделирование.

Это правильный продуктовый подход, потому что он даёт:
- управляемую креативность
- чистый UX
- предсказуемую производительность
- более простую валидацию ассетов
- меньшее количество «сломанных» комбинаций
- более быстрые итерации контент-паков

Система должна строиться на основе **типизированных слотов** с ограничениями.

### Предлагаемые слоты куклы ()
- голова
- волосы
- шляпа
- рога
- нимб
- оболочка тела
- внутренняя вставка
- воротник
- крылья
- левая рука
- правая рука
- левый рукав
- правый рукав
- нижняя часть тела
- ноги / основание
- хвост
- опциональные декоративные элементы
- фон
- передний план
- плавающие пропсы
- музыкальный трек

## Модель кастомизации
Каждый слот должен поддерживать управляемую кастомизацию через метаданные.

### Метаданные на уровне слота должны определять:
- допустимые asset ID
- transform по умолчанию
- минимальный / максимальный scale
- минимальный / максимальный rotation
- anchor point
- границы коллизий
- зависимости
- исключения
- опциональные флаги совместимости с анимацией

### Ключевой принцип дизайна
Пользователь должен получать широкие возможности кастомизации, но только внутри контролируемых эстетических и технических рамок.

Это означает:
- никакого полностью свободного размещения для основных частей тела
- привязку ассетов к anchor points
- ограниченные transform-параметры
- правила совместимости между частями
- визуальная целостность поддерживается на уровне системы

## Зоны ответственности бэкенда
Go-бэкенд должен обеспечивать:
- аутентификацию
- работу с гостевыми сессиями
- сценарий апгрейда аккаунта
- доставку каталога
- доставку asset manifest
- сохранение / загрузку проектов
- версионируемое хранение проектов
- проверку entitlement
- ingestion аналитики
- feature flags / remote config
- админские инструменты модерации и публикации контента

### Что хранить в PostgreSQL:
- пользователей
- связи guest-to-account
- проекты кукол
- версии проектов
- метаданные каталога
- правила совместимости слотов
- покупки и entitlements
- историю шеринга
- аналитические факты, которым нужна долговременная сохранность
- назначения экспериментов
- админские записи по контенту

### Что хранить в Redis:
- горячие manifest-файлы и каталог
- rate limits
- кэш feature flags
- временные блокировки
- состояние задач
- состояние экспорта / рендера
- краткоживущие project/session state
- счётчики и буферы эфемерной аналитики

## Требования к пайплайну 3D-ассетов
Исходные 3D-сканы должны пройти полноценный production-пайплайн, прежде чем их можно будет использовать в приложении.

### Обязательные шаги
- очистка меша
- ретопология или упрощение
- очистка UV
- оптимизация / сжатие текстур
- нормализация pivot / origin
- нормализация масштаба
- расстановка anchor points
- генерация collision mesh
- генерация LOD при необходимости
- генерация preview thumbnails
- генерация метаданных
- версионирование

### Для каждого ассета нужны метаданные:
- asset ID
- слот / категория
- отображаемое имя
- версия
- URL превью-изображения
- URL 3D-файла
- ссылки на текстуры
- bounding box
- transform по умолчанию
- допустимый диапазон transform
- совместимые слоты / ассеты
- исключённые ассеты
- правила зависимостей
- флаги поддержки анимации

## Стратегия рендера
### Рекомендуемый вариант для V1
Сначала использовать **рендер на устройстве**.

Преимущества:
- ниже инфраструктурные затраты
- быстрее обратная связь для пользователя
- меньше узких мест на бэкенде
- проще архитектура MVP

### Откладывать серверный рендер до тех пор, пока не появится одно из условий:
- понадобится премиальный экспорт в высоком разрешении
- видео станут длиннее или тяжелее
- старые устройства начнут испытывать трудности
- понадобятся премиальные визуальные эффекты
- потребуется детерминированно одинаковый рендер на всех устройствах

### Рекомендация
- **V1:** локальный рендер / экспорт на устройстве
- **Позже:** опциональный облачный премиальный рендер

---

## Стратегия шеринга
Считать Instagram важным направлением шеринга, но не единственным способом публикации.

### Рекомендуемый подход
- генерировать видео локально
- открывать нативный iOS share sheet
- позволять пользователю делиться туда, куда это поддерживается системой

Это делает продукт гибким и не привязывает его слишком жёстко к одному каналу распространения.

---

## Стратегия монетизации
### Рекомендуемая модель
- первый успешный экспорт / шеринг — бесплатно
- после первого успешного экспорта / шеринга показывается пейволл
- платный доступ открывает:
  - неограниченные экспорты
  - премиальные наборы ассетов
  - премиальную музыку
  - премиальные сцены / фоны
  - опциональный HD-рендер

### Платёжный стек
- StoreKit 2 на устройстве
- серверная синхронизация / проверка entitlement на бэкенде
- поддержка восстановления покупок

## Основные решения, которые ещё нужно принять
1. **Только RealityKit или гибридный рендеринг-стек**
   - Рекомендация: RealityKit-first

2. **Рендер на устройстве или в облаке**
   - Рекомендация: на устройстве для V1

3. **Anonymous-first или account-first онбординг**
   - Рекомендация: сначала анонимный сценарий, апгрейд позже

4. **Насколько свободно разрешать трансформации**
   - Рекомендация: ограниченные transform-параметры на уровень слота

5. **Одна базовая схема куклы или несколько семейств шаблонов**
   - Рекомендация: одна базовая схема, семейства добавить позже

6. **Сколько ассетов поставлять в бандле приложения, а сколько скачивать удалённо**
   - Рекомендация: в приложение включать только стартовый набор, остальное доставлять удалённо

7. **Детали монетизационной модели**
   - Рекомендация: первый экспорт бесплатно, дальше подписка или unlock + packs

8. **Объём админского инструментария**
   - Рекомендация: лёгкая внутренняя CMS / dashboard с раннего этапа

9. **Глубина аналитики**
   - Рекомендация: инструментировать каждый ключевой шаг и взаимодействие с ассетами

10. **Стратегия масштабирования бэкенда**
   - Рекомендация: сначала модульный монолит, разделять только при реальной необходимости

## Предлагаемый поэтапный roadmap

### Фаза 1: Прототип
- один шаблон куклы
- 3–5 слотов
- 20–30 ассетов
- простые ограничения transform
- одна сцена
- один музыкальный трек
- локальный рендер / экспорт
- нативный share flow
- простой триггер пейволла

### Фаза 2: MVP
- полная slot taxonomy
- удалённая доставка каталога
- сохранённые проекты
- аналитика
- интеграция StoreKit 2
- entitlement backend
- админский контент-пайплайн
- оптимизация по сбоям и памяти
- hooks для экспериментов

### Фаза 3: Масштабирование
- премиальные наборы ассетов
- сезонные контент-дропы
- больше шаблонов сцен
- опциональный облачный рендер
- эксперименты с онбордингом и пейволлом
- социальный / галерейный слой при необходимости

## Риски, которые нужно заранее учитывать
- качество 3D-сканов может быть неравномерным и потребовать серьёзной доработки
- слишком большая свобода кастомизации может разрушить визуальное качество
- потребление памяти на мобильных устройствах может резко вырасти из-за больших текстур и мешей
- время экспорта / рендера может быть слишком большим на старых устройствах
- правила совместимости каталога могут стать трудноуправляемыми без сильной системы метаданных
- UX монетизации должен быть очень аккуратно спроектирован под требования App Store
- ожидания по шеринг-флоу могут быстро перерасти первый экспортный пайплайн
