# Исследование веб-приложения — Конструктор кукол в музыкальной шкатулке

---

## Краткое описание проекта

Данный документ содержит исследование и рекомендации по разработке **веб-приложения**, которое позволяет пользователям создавать кастомную виртуальную куклу, помещать её в стилизованную сцену музыкальной шкатулки, добавлять декоративные элементы и музыку, рендерить короткое видео и делиться им.

Основная целевая аудитория — **пользователи iPhone с Chrome или Safari**, вместо изначально планировавшегося нативного iOS-приложения создаётся мобильно-ориентированный веб-опыт, работающий на том же мощном оборудовании Apple. Веб-приложение дополнительно охватывает пользователей Android и десктопа с нулевыми затратами на дистрибуцию.

Целевой пользовательский флоу не изменился:
1. Анимированный лендинг
2. Пошаговый конструктор куклы
3. Режим глобальной корректировки с ограничениями
4. Декорирование сцены и стилизация
5. Выбор музыки
6. Локальный рендер/экспорт видео
7. Флоу шеринга
8. Пейволл после первого успешного шеринга/экспорта

## Почему веб вместо нативного iOS

| Фактор | Нативный iOS | Веб-приложение |
|---|---|---|
| Дистрибуция | Только App Store; ревью, задержки | Мгновенно по ссылке; без гейткиперов |
| Комиссия платформы | 30% комиссия Apple (15% для малого бизнеса) | 0% комиссия платформы при прямых платежах (Stripe и т.д.) |
| Охват | Только iPhone | iPhone, Android, десктоп — любой современный браузер |
| Обнаруживаемость | Поиск в App Store + ASO | SEO, ссылки из соцсетей, прямые URL |
| Цикл обновлений | Ревью в App Store на каждый релиз | Деплой мгновенно, без ревью |
| Порог входа | Скачать → установить → открыть | Нажать на ссылку → пользоваться сразу |
| 3D-возможности | RealityKit (отлично) | Three.js / WebGPU (очень хорошо, с некоторыми ограничениями) |
| Экспорт видео | AVFoundation (нативный, быстрый) | WebCodecs / MediaRecorder / ffmpeg.wasm (рабочий вариант, сложнее) |
| Офлайн | Полный нативный офлайн | Service Workers / PWA (хорошо, но с ограничениями) |
| Доступ к API устройства | Полный (хаптика, ARKit и т.д.) | Ограниченный (Web Share API, вибрация, ограниченный доступ к файлам) |
| Платежи | Только StoreKit 2 | Stripe, PayPal, любой веб-провайдер платежей |

### Ключевые преимущества web-first подхода

- **Нет 30% налога Apple** — прямая обработка платежей сохраняет значительно больше выручки
- **Нулевой порог входа** — пользователи нажимают на ссылку и начинают собирать куклу; критично для вирусных/социальных петель
- **Мгновенные обновления** — выкатываем фиксы и новый контент без ревью App Store
- **Кроссплатформенность с первого дня** — Android-пользователи доступны сразу
- **Шеринг через ссылки** — каждый шер может содержать прямую ссылку обратно в приложение
- **SEO и контент-маркетинг** — страницы галереи, лендинги, профили создателей индексируются поисковиками

### Ключевые риски web-first подхода

- **Потолок производительности 3D-рендеринга** — WebGL в мобильном Safari имеет известные ограничения; поддержка WebGPU ещё на ранней стадии
- **Экспорт видео сложнее** — нет нативного AVFoundation; нужно использовать WebCodecs, MediaRecorder или ffmpeg.wasm
- **Особенности мобильного Safari** — ограничения автовоспроизведения аудио, ограничения PWA, непоследовательное поведение полноэкранного режима
- **Нет пуш-уведомлений (Safari)** — ограниченное вовлечение на iOS по сравнению с нативом
- **Ограничения хранилища** — квоты IndexedDB в мобильных браузерах; менее надёжно, чем нативное хранение
- **Нет присутствия в App Store** — теряется канал обнаружения «просмотр магазина» (компенсируется соцсетями/SEO)

## Рекомендуемый технологический стек

### Фронтенд (веб-приложение)

- **React 19** с **Next.js 15 (App Router)** — SSR для лендинга/маркетинга, CSR для редактора
- **TypeScript** — строгий режим повсеместно
- **Three.js** через **React Three Fiber (R3F)** — 3D-композиция сцены, привязка к слотам, ограниченные трансформации, рендеринг на экране
- **Drei** — утилитарная библиотека R3F (контролы орбиты, загрузчики, хелперы)
- **glTF / GLB** — стандартный веб-формат 3D (заменяет USDZ)
- **Zustand** — легковесное глобальное состояние для состояния композиции редактора
- **Tailwind CSS** — утилитарная стилизация, mobile-first адаптивный дизайн
- **Framer Motion** — анимация лендинга и UI-переходы
- **Howler.js** — кроссбраузерное воспроизведение аудио с обходом ограничений мобильного Safari
- **IndexedDB** (через обёртку **idb**) — локальное сохранение черновиков проектов
- **Web Share API** — нативно ощущаемый шаринг на мобильных
- **Service Worker / PWA-манифест** — офлайн-кеширование ассетов, добавление на домашний экран

### Стратегия экспорта видео (критический путь)

Экспорт видео в вебе — самая сложная техническая задача. Три жизнеспособных подхода в порядке приоритета:

#### Вариант А: WebCodecs API + MP4 Muxer (рекомендуется для V1)
- Использовать `WebCodecs` (VideoEncoder) для кодирования фреймов, захваченных с Three.js canvas
- Мультиплексировать закодированные фреймы + аудио в MP4 через библиотеку типа **mp4-muxer** или **webm-muxer**
- **Плюсы:** быстро, работает на устройстве, не нужен сервер, аппаратное ускорение кодирования
- **Минусы:** WebCodecs не поддерживается в Safari < 16.4 (iOS 16.4+); нужен фоллбэк
- **Поддержка браузерами:** Chrome (отлично), Safari 16.4+ (хорошо), Firefox (частично)

#### Вариант Б: MediaRecorder API (фоллбэк)
- Использовать `MediaRecorder` для захвата стрима с `canvas.captureStream()`
- На выходе WebM (Chrome) или MP4 (Safari)
- **Плюсы:** более простой API, более широкая поддержка
- **Минусы:** меньше контроля над качеством/битрейтом; WebM требует конвертации для Instagram; синхронизация аудио может быть проблемой

#### Вариант В: Серверный рендер (отложить)
- Отправить состояние композиции на бэкенд; рендерить headless Three.js (или ffmpeg) на сервере
- Вернуть MP4 клиенту
- **Плюсы:** детерминированный результат, работает в любом браузере, возможно более высокое качество
- **Минусы:** требуется серверная инфраструктура, задержки, стоимость за рендер, управление очередями

#### Рекомендация
- **V1:** WebCodecs с фоллбэком на MediaRecorder для неподдерживаемых браузеров
- **Позже:** опциональный серверный рендер для премиум HD-экспорта или старых устройств
- **Критично:** тщательно тестировать экспорт видео на iPhone Safari и Chrome; это зона наивысшего риска

### Бэкенд

- **Golang** — API и бэкенд-сервисы (то же, что в iOS-плане)
- **Redis** — кеш, очереди, рейт-лимиты, горячее состояние, короткоживущие сессионные данные

- **Docker** — упаковка
- **GitHub Actions** — CI/CD
- **OpenTelemetry** — наблюдаемость

### Платежи

- **Stripe Checkout** или **Stripe Elements** — встроенный платёжный UI

- внутренняя вставка

- крылья
- левая рука
- правая рука
- левый рукав
- правый рукав
- нижняя часть
- ноги/основание
- хвост
- дополнительные декоративные элементы
- фон
- передний план
- парящие декорации
- музыкальный трек

## Модель кастомизации

Идентична iOS-дизайну. Каждый слот поддерживает контролируемую кастомизацию через метаданные.

### Метаданные для каждого слота должны определять:
- допустимые ID ассетов
- дефолтную трансформацию
- мин/макс масштаб
- мин/макс вращение
- точку привязки (anchor)
- границы коллизии
- зависимости
- исключения
- опциональные флаги совместимости анимаций

### Принцип дизайна
Пользователи должны иметь широкие возможности кастомизации, но только в рамках контролируемой эстетической и технической оболочки.

Это означает:
- нет свободного позиционирования основных частей тела
- привязка ассетов к якорным точкам
- ограниченные трансформации
- правила совместимости между деталями
- визуальная целостность сохраняется по дизайну

## 3D-формат и пайплайн ассетов

### Формат: glTF 2.0 / GLB

USDZ — формат только для экосистемы Apple. Для веба стандартом является **glTF 2.0** (бинарная форма: GLB).

- Универсальная поддержка Three.js, Babylon.js и всеми основными веб 3D-движками
- Компактный бинарный формат со встроенными текстурами
- Поддерживает PBR-материалы, анимации, морф-таргеты
- Отличные инструменты: Blender экспортирует нативно, glTF-Transform для оптимизации
- При необходимости может быть сконвертирован в USDZ позже, если нативное iOS-приложение будет создаваться

### Необходимые шаги пайплайна ассетов
- Очистка меша
- Ретопология или упрощение (цель: 5k–15k треугольников на деталь для мобильного веба)
- Очистка UV
- Оптимизация текстур: KTX2-сжатые текстуры через Basis Universal (критично для веб-производительности)
- Draco-сжатие мешей для GLB-файлов
- Нормализация пивота/начала координат
- Нормализация масштаба
- Размещение якорных точек
- Генерация коллизионных мешей (упрощённые ограничивающие объёмы)
- Генерация LOD при необходимости
- Генерация превью-миниатюр (формат WebP)
- Генерация метаданных (JSON-сопутствующий файл или встроено в glTF extras)
- Версионирование

### Веб-специфические оптимизации
- **KTX2 + Basis Universal** сжатие текстур — в 4–8 раз меньше PNG/JPEG, декомпрессия на GPU
- **Draco** сжатие мешей — сокращение геометрических данных на 80–90%
- **Прогрессивная загрузка** — показать низкокачественное превью, стримить полное качество
- **Агрессивное CDN-кеширование** — иммутабельные URL ассетов с хешами контента
- Цель — **< 500 КБ на GLB-ассет** после сжатия
- Цель — **< 5 МБ на начальную загрузку сцены** (стартовые ассеты)

### Каждый ассет должен включать метаданные:
- ID ассета
- слот/категория
- отображаемое имя
- версия
- URL превью (WebP)
- URL GLB-файла
- ссылки на текстуры (KTX2)
- ограничивающий прямоугольник (bounding box)
- дефолтная трансформация
- допустимый диапазон трансформаций
- совместимые слоты/ассеты
- исключённые ассеты
- правила зависимостей
- флаги поддержки анимации
- размер файла (для UI загрузки)
- количество треугольников (для бюджетирования производительности)

В основном те же, что в iOS-плане, с дополнениями для веб-специфических нужд.

Go-бэкенд должен обеспечивать:

- сохранение/загрузку проектов (облачная синхронизация, не только локальная)
- версионированное хранение проектов
- проверку прав доступа (интеграция с Stripe webhooks)
- создание Stripe Checkout сессий и верификацию платежей
- приём аналитики
- фича-флаги / удалённая конфигурация
- инструменты модерации и публикации для админов
- CORS-конфигурацию для фронтенда
- рейт-лимитирование для API и эндпоинтов ассетов

### PostgreSQL должен хранить:
- пользователей
- маппинг гость → аккаунт
- проекты кукол
- версии проектов
- метаданные каталога
- правила совместимости слотов
- покупки и права доступа (ссылки на Stripe customer/payment)
- историю шеринга
- аналитические факты, требующие долговременного хранения
- назначения экспериментов
- записи админского контента

### Redis должен хранить:
- горячий каталог/манифесты
- рейт-лимиты
- кеш фича-флагов
- временные блокировки
- состояние задач (при добавлении серверного рендера)
- состояние экспорта/рендера
- короткоживущее состояние сессий
- счётчики и эфемерные буферы аналитики

## Стратегия рендеринга

### V1: На устройстве (в браузере)

Преимущества:
- ниже стоимость инфраструктуры
- быстрее для пользователя
- меньше узких мест на бэкенде
- проще MVP-архитектура
- работает офлайн после загрузки ассетов

### Подход
1. Three.js рендерит скомпонованную сцену с анимацией вращения на 360°
2. Фреймы захватываются с WebGL canvas через `canvas.toDataURL()` или `OffscreenCanvas`
3. Фреймы кодируются через WebCodecs VideoEncoder (H.264)
4. Аудиотрек миксуется через Web Audio API
5. Мультиплексирование в MP4 через mp4-muxer
6. Результирующий Blob предлагается для скачивания или шеринга

### Целевые показатели производительности
- 1080×1920 при 30 fps
- 10 секунд по умолчанию = 300 фреймов
- Целевое время рендера: < 30 секунд на iPhone 12+ (Chrome)
- Целевое время рендера: < 60 секунд на iPhone 12+ (Safari, из-за различий WebCodecs)

### Фоллбэк
- MediaRecorder для браузеров без поддержки WebCodecs
- Уменьшенное разрешение (720×1280) для старых устройств

### Отложить серверный рендеринг до:
- необходимости премиум-экспорта высокого разрешения
- увеличения длительности/тяжести видео
- блокирующих ограничений браузера
- необходимости премиум-визуальных эффектов
- необходимости детерминированного одинакового рендера на всех устройствах

## Стратегия шеринга

### Web Share API (основной)
- `navigator.share()` с поддержкой файлов доступен в мобильном Chrome и Safari
- Позволяет шерить MP4-файл напрямую, в том числе в Instagram, WhatsApp, Telegram и т.д.
- Фоллбэк на скачивание, если Web Share API недоступен (десктопные браузеры)

### Флоу шеринга
1. Видео рендерится локально в браузере
2. На мобильном: `navigator.share({ files: [mp4File], title: '...' })` — открывает нативный шеринг
3. На десктопе: кнопка скачивания + ссылка на галерею (Фаза 2)
4. Опционально: генерация шарeable ссылки на галерею (требует облачного сохранения проекта)

### Важно
- Instagram не на всех устройствах надёжно принимает share intent с видео; тестировать тщательно
- Фоллбэк: сохранить видео в фотоплёнку (через скачивание), пользователь постит вручную
- Рассмотреть добавление кнопки «Сохранить в Фото» как основное действие наряду с шерингом

- Серверная обработка вебхуков для управления правами
- Проверка прав перед экспортом (API-вызов к бэкенду)
- Гостевые пользователи: права хранятся на сервере по анонимному UUID сессии
- Пользователи с аккаунтом: права привязаны к аккаунту
- Восстановление: логин в аккаунт восстанавливает все права

### Сравнение выручки
| | Нативный iOS | Веб-приложение |
|---|---|---|
| Покупка за $9.99 | $6.99 после комиссии Apple | ~$9.41 после комиссии Stripe |
| Подписка $4.99/мес | $3.49 после комиссии Apple | ~$4.54 после комиссии Stripe |

Веб-путь сохраняет ~35% больше выручки с каждой транзакции.

## PWA (Progressive Web App): соображения

### Рекомендуется для Фазы 1
- Добавить `manifest.json` с именем приложения, иконками, цветом темы
- Зарегистрировать Service Worker для кеширования ассетов
- Включить запрос «Добавить на домашний экран»
- Кешировать все загруженные GLB-ассеты для офлайн-редактирования

### Ограничения PWA на iOS Safari
- Нет пуш-уведомлений (поддерживаются только с iOS 16.4+, ненадёжно)
- Хранилище может быть очищено, если PWA не используется какое-то время
- Нет фоновой синхронизации
- Ограничение 50 МБ в некоторых контекстах без взаимодействия пользователя
- Полноэкранный режим имеет проблемы с «чёлкой» и безопасными зонами

### Рекомендация
- Относиться к PWA как к «приятному бонусу», а не основному требованию
- Сначала проектировать как обычное мобильное веб-приложение; PWA-фичи — прогрессивное улучшение
- Не полагаться только на офлайн; предполагать наличие сети для первой загрузки и доставки ассетов

## Требования к производительности мобильного веба

### Целевые устройства
- Основные: iPhone 12 и новее, Chrome или Safari
- Второстепенные: iPhone 11, современные Android-флагманы (Chrome)
- Минимальные: любое устройство с поддержкой WebGL 2.0

### Бюджеты производительности
- **Начальная загрузка страницы (лендинг):** < 3 секунд на 4G (3D-ассеты ещё не загружены)
- **Время до интерактивного редактора:** < 5 секунд (после нажатия «Создать»)
- **3D-сцена с 5 ассетами:** 30+ fps при орбите/манипуляции
- **Общий бандл ассетов (Фаза 1):** < 20 МБ (сжатый, доставляемый через CDN)
- **Отдельный GLB-ассет:** < 500 КБ
- **Использование памяти:** < 300 МБ при редактировании (чтобы избежать вытеснения вкладки Safari)

### Стратегии оптимизации
- Ленивая загрузка 3D-движка только при входе в редактор (code splitting)
- Прогрессивная загрузка ассетов — сначала низкокачественные миниатюры, GLB подгружается при выборе
- KTX2-текстуры для GPU-сжатой доставки
- Draco-сжатые меши
- Пулинг и утилизация Three.js объектов
- Мониторинг и ограничение размеров текстурных атласов
- Использование `OffscreenCanvas` где поддерживается для пайплайна рендера

## Подводные камни мобильных браузеров

### Safari-специфичное
- **Автовоспроизведение аудио заблокировано** — необходимо начинать аудио по жесту пользователя; Howler.js решает это
- **Ограничения памяти WebGL** — Safari агрессивно убивает WebGL-контексты; мониторить и снижать потребление
- **Нет WebCodecs до 16.4** — нужен фоллбэк на MediaRecorder
- **IndexedDB может быть очищена** — никогда не считать её надёжным хранилищем; важное состояние синхронизировать на сервер
- **Баг 100vh** — использовать `dvh` (dynamic viewport height) или JavaScript-измерение
- **Нет Badging API** — нельзя показывать бейджи уведомлений

### Chrome на iOS
- Chrome на iOS использует WebKit «под капотом» (политика Apple) — ведёт себя как Safari для WebGL, WebCodecs и т.д.
- Это значит: Chrome на iPhone = движок рендеринга Safari WebKit с UI Chrome
- Все ограничения Safari применимы к Chrome на iOS

### Следствие
- **На iPhone нет способа обойти ограничения WebKit** — все браузеры используют один движок
- Тестировать в первую очередь в Safari; поведение Chrome на iOS будет почти идентичным
- Для Android/десктопа Chrome V8/Blink предлагает лучшую поддержку WebCodecs и WebGPU

1. **Только Three.js или добавить Babylon.js как альтернативу**
   - Рекомендация: Three.js через React Three Fiber (больше экосистема, лучшая интеграция с React)

2. **Рендер на устройстве или в облаке**
   - Рекомендация: на устройстве для V1 (WebCodecs + фоллбэк MediaRecorder)

3. **Сначала анонимный или сначала аккаунт при онбординге**
   - Рекомендация: сначала анонимный, апгрейд через email magic link позже

   - Рекомендация: ограниченные трансформации по слотам (без изменений к iOS-плану)

5. **Одна базовая схема куклы или несколько семейств шаблонов**
   - Рекомендация: одна базовая схема с опциональными семействами позже

   - Рекомендация: доставлять ассеты по запросу по слотам; агрессивно кешировать через Service Worker

    - Рекомендация: модульный монолит сначала, разделять только при необходимости

11. **Сроки внедрения WebGPU**
    - Рекомендация: строить на WebGL 2.0 для V1; внедрять WebGPU когда поддержка Safari стабилизируется (мониторить)

    - Рекомендация: целиться в H.264 в MP4 (максимальная совместимость для шеринга в соцсетях); WebM только как промежуточный формат

## Риски, которые нужно учитывать

- **Экспорт видео в Safari — элемент наивысшего риска** — поддержка WebCodecs относительно новая; вывод MediaRecorder может быть не идеальным для Instagram; необходимо тщательное тестирование
- **Chrome на iOS = WebKit** — нет способа получить лучшую 3D- или видео-производительность на iPhone сменой браузера; все браузеры на iOS используют один движок
- **Качество 3D-сканов** может быть нестабильным и потребовать значительной доработки (риск без изменений)
- **Давление памяти мобильного WebGL** — Safari агрессивно вытесняет WebGL-контексты; необходимо мониторить и управлять аккуратно
- **Избыточная свобода кастомизации** может сломать визуальное качество (риск без изменений)
- **Память мобильных текстур** может резко вырасти с большими текстурами; KTX2-сжатие критически важно
- **IndexedDB ненадёжна** на iOS — данные могут быть очищены; важные проекты нуждаются в серверном хранении
- **UX оплаты через Stripe на мобильном** должен быть чистым и нативно ощущаемым; плохой чекаут = брошенные покупки
- **Правила совместимости каталога** могут стать трудноуправляемыми без сильного инструментария метаданных (риск без изменений)
- **Трение при шеринге в соцсети** — если Web Share API ненадёжно передаёт видеофайлы на целевые платформы, фоллбэк-флоу должен быть отполирован
- **Адопция PWA «Добавить на домашний экран»** низкая; не стоит переинвестировать сюда

## Веб vs iOS: что проще, что сложнее

### Проще в вебе
- Дистрибуция и вирусные петли (просто поделиться URL)
- Экономика платежей (Stripe ~3% vs Apple 15–30%)
- Обновления и деплой контента (мгновенно, без ревью)
- Кроссплатформенность (Android и десктоп бесплатно)
- A/B-тестирование и эксперименты
- Аналитика и инструментирование воронки
- SEO и контент-маркетинг

### Сложнее в вебе
- Пайплайн экспорта видео (нет аналога AVFoundation)
- Потолок производительности 3D-рендеринга (WebGL vs Metal/RealityKit)
- Работа с аудио в мобильном Safari (ограничения автовоспроизведения)
- Постоянное локальное хранилище (IndexedDB ненадёжна)
- Пуш-уведомления для вовлечения (ограничены на iOS)
- «Нативное» ощущение приложения (возможно с хорошей PWA-работой, но никогда полностью нативное)
- Доступ к возможностям устройства (хаптика, AR и т.д.)
